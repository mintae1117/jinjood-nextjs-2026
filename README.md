# 진주떡집 (Jinjood) - Next.js

부산 수영구 남천동에 위치한 전통 떡집 웹사이트

- **Production**: https://jinjood.com
- **Staging**: https://jinjood-nextjs.vercel.app (배포 후 생성)

---

## 결제 기능 도입 종합 가이드

> 결제 기능을 도입하기 위해 필요한 **법적 준비사항**, **기술 설계**, **사업 정책 확정 사항**을 정리한 문서입니다.
> 개발 착수 전 사업 정책이 먼저 확정되어야 합니다.

### 1. 사업자 등록 / 인허가 (법적 필수 - 최우선)

#### 1-1. 통신판매업 신고

- **현재 상태**: 사업자등록번호(452-23-00331)는 있으나, **통신판매업 신고** 여부 확인 필요
- **필수 조건**: 온라인으로 상품을 판매하려면 관할 구청(수영구청)에 통신판매업 신고 필수
- **준비물**:
  - 사업자등록증 사본
  - 구매안전서비스 이용확인증 (PG사 계약 후 발급)
  - 도메인 등록 확인
- **미신고 시**: 과태료 최대 1,000만원, 영업정지 가능
- **소요 시간**: 신청 후 약 3~7일

#### 1-2. 식품 관련 인허가 확인

- 이미 오프라인 매장을 운영 중이므로 **식품제조가공업** 또는 **즉석판매제조가공업** 허가가 있을 것
- 온라인 판매 시 별도 추가 허가가 필요한지 관할 보건소에 확인
- **식품 표시 기준**: 온라인 판매 시 아래 항목을 상품 페이지에 표시 의무
  - 원재료명
  - 알레르기 유발물질
  - 유통기한
  - 보관방법

#### 1-3. 구매안전서비스 가입 (에스크로)

- 전자상거래법상 **연매출 기준에 따라** 에스크로 또는 소비자피해보상보험 가입 필수
- PG사(토스페이먼츠 등) 계약 시 자동으로 충족되는 경우가 많음

### 2. PG사(결제대행사) 선정 및 계약

#### 2-1. PG사 비교

| 항목 | 토스페이먼츠 | NHN KCP | 아임포트(포트원) |
|------|------------|---------|--------------|
| **연동 난이도** | 쉬움 (SDK 제공) | 보통 | 쉬움 (통합 SDK) |
| **수수료** | 카드 2.5~3.5% | 카드 2.5~3.5% | PG사 수수료 + 추가 없음 |
| **지원 결제** | 카드, 계좌이체, 가상계좌, 카카오페이, 네이버페이, 토스페이 | 카드, 계좌이체, 가상계좌, 카카오페이 | 여러 PG사 통합 |
| **정산 주기** | D+2~5 영업일 | D+3~5 영업일 | PG사에 따름 |
| **최소 요건** | 사업자등록 + 통신판매업 | 사업자등록 + 통신판매업 | 동일 |

**추천**: 토스페이먼츠 - Next.js SDK 지원이 좋고, 문서가 한국어로 잘 되어 있음

#### 2-2. PG사 계약 시 준비 서류

- 사업자등록증 사본
- 통신판매업 신고증
- 대표자 신분증
- 사이트 URL (실제 동작하는 상태여야 함)
- 판매 상품 설명 (식품 카테고리)
- 정산 계좌 정보 (부산은행 112-2038-7604-08 / 정은아)

#### 2-3. 심사 및 소요 기간

- PG사 심사: 보통 **3~7 영업일**
- 식품 판매의 경우 추가 서류 요청 가능 (식품 인허가 관련)

### 3. 법적 문서 업데이트

#### 3-1. 이용약관 업데이트 필요 사항

현재 `src/components/legal/TermsContent.tsx`에 결제 관련 조항(제8~11조)이 이미 존재하지만, 실제 결제 도입 시 구체화 필요:

- **제9조 (결제방법)**: 실제 사용할 PG사와 결제 수단 명시
- **제10조 (배송)**: 배송비 정책 구체화 (현재 3,000원 고정이면 명시)
- **제11조 (환불/교환)**: 환불 처리 절차와 소요 기간 구체화
- **추가 필요**: 주문 취소 가능 시점 (결제 후 ~ 배송 전), 부분 환불 정책

#### 3-2. 개인정보처리방침 업데이트 필요 사항

현재 `src/components/legal/PrivacyContent.tsx`에 기본 구조는 있으나:

- **수집 항목 추가**: 배송지 주소, 수령인 이름, 수령인 연락처
- **위탁업체 구체화**: 실제 계약한 PG사명, 배송업체명 기재 필수
  - 예: "토스페이먼츠 주식회사 - 결제 처리"
  - 예: "CJ대한통운 - 배송 업무"
- **보유기간**: 전자상거래법에 따라 결제 기록 5년, 배송 기록 5년 보관 의무 명시

#### 3-3. 결제 시 필수 고지 사항 (전자상거래법)

결제 페이지에 반드시 표시해야 할 내용:

- 상품명, 수량, 가격
- 배송비
- 총 결제 금액
- 결제 수단
- **"구매 조건 확인 및 결제에 동의합니다" 체크박스** (필수)
- 환불/교환 불가 사유 고지 (식품 특성)
- 주문자 정보와 수령인 정보

#### 4-2. RLS 정책

- 사용자는 본인의 주문만 조회 가능
- 주문 생성은 인증된 사용자만
- 주문 상태 변경은 서버 측(service role)에서만

#### 4-3. 중요: 상품 스냅샷

- `order_items`에 주문 시점의 **상품명과 가격을 복사 저장**
- 나중에 상품 가격이 변경되더라도 주문 기록은 유지되어야 함

### 5. 결제 플로우 설계

```
[장바구니] → [주문서 작성] → [결제 요청] → [PG사 결제창] → [결제 완료] → [주문 확인]
                                    ↓                              ↓
                               [결제 실패]                    [결제 승인 API]
                                    ↓                         (서버 검증)
                               [에러 안내]                         ↓
                                                          [주문 상태 업데이트]
                                                                  ↓
                                                          [장바구니 비우기]
```

#### 5-1. 상세 단계

**Step 1: 주문서 작성 페이지** (`/checkout`)

- 장바구니에서 넘어온 상품 목록 표시
- 배송지 입력 (우편번호 검색 API - 다음 주소 검색)
- 수령인 정보 입력
- 배송 메모 선택/입력
- 결제 수단 선택
- 최종 금액 확인
- 약관 동의 체크박스

**Step 2: 결제 요청** (클라이언트)

- 주문 정보를 서버에 보내서 `orders` 레코드 생성 (status: pending)
- PG사 SDK로 결제창 호출

**Step 3: 결제 승인** (서버 - Next.js API Route 또는 Server Action)

- PG사로부터 받은 `paymentKey`, `orderId`, `amount`를 서버에서 검증
- 토스페이먼츠 승인 API 호출 (`POST /v1/payments/confirm`)
- **금액 위변조 검증 필수**: DB에 저장된 주문 금액과 PG사 응답 금액을 서버에서 비교
- 성공 시: `orders.status = 'paid'`, `payments` 레코드 생성
- 실패 시: `orders.status = 'failed'`, 에러 로깅

**Step 4: 결제 완료 페이지** (`/checkout/success`)

- 주문번호, 결제 금액, 배송 예정 안내
- 주문 내역 페이지 링크

**Step 5: 결제 실패 페이지** (`/checkout/fail`)

- 실패 사유 안내
- 재시도 또는 장바구니로 돌아가기

### 7. 보안 체크리스트

- [ ] **결제 금액 서버 검증**: 클라이언트가 보낸 금액을 절대 신뢰하지 않고, DB의 주문 금액과 PG사 응답 금액을 서버에서 비교
- [ ] **PG사 Secret Key**: 환경변수로만 관리, 클라이언트에 절대 노출 금지 (`TOSS_SECRET_KEY` - `NEXT_PUBLIC_` 접두사 없이)
- [ ] **웹훅 검증**: PG사 웹훅 호출 시 서명(signature) 검증
- [ ] **HTTPS 필수**: Vercel 배포 시 자동 적용
- [ ] **카드번호 비저장**: PG사 SDK가 처리하므로 직접 수집/저장 절대 금지 (PCI-DSS)
- [ ] **SQL Injection 방지**: Supabase 클라이언트 사용으로 기본 방어됨
- [ ] **주문 상태 변조 방지**: 클라이언트에서 직접 status 변경 불가하도록 RLS 설정

### 8. 세금 / 회계 관련

#### 8-1. 현금영수증

- 현금성 결제(계좌이체, 가상계좌) 시 **현금영수증 발행 의무** (소비자 요청 시)
- PG사에서 현금영수증 발행 기능을 제공하는 경우가 많음
- 미발행 시 가산세 부과 가능

#### 8-2. 세금계산서

- B2B 거래 시 세금계산서 발행 필요 (일반 소비자 대상이면 해당 없을 수 있음)
- 대량 주문(답례떡, 이바지떡) 시 사업자 고객이 요청할 수 있음

#### 8-3. 매출 신고

- PG사를 통한 결제는 자동으로 국세청에 보고됨
- 부가가치세 신고 시 PG사 매출 자료 활용

### 9. 배송 관련 고려사항

#### 9-1. 떡류 특수성

- **신선식품**이므로 당일/익일 배송 필수
- 배송 가능 지역 제한 필요 (전국 택배 가능한지, 부산 인근만 가능한지)
- 여름철 아이스팩/보냉 포장 고려
- **주문 마감 시간** 설정 (예: 당일 배송은 오전 10시까지 주문)

#### 9-2. 배송비 정책 확정 필요

- 현재 고정 3,000원으로 설정됨
- 지역별 차등 배송비 (제주/도서산간 추가)
- 퀵배송(부산 지역) vs 택배(타 지역) 구분 여부

#### 9-3. 택배사 연동 (선택)

- 초기에는 수동 운송장 입력으로 시작 가능
- 추후 CJ대한통운/롯데택배 API 연동 고려

### 10. 구현 우선순위 로드맵

#### Phase A: 사전 준비 (개발 전 - 1~2주)

1. 통신판매업 신고 확인/진행
2. PG사(토스페이먼츠) 가입 신청 및 심사
3. 배송비 정책, 배송 가능 지역, 주문 마감 시간 등 **사업 정책 확정**
4. 택배사 선정

#### Phase B: 기반 구축 (1주)

1. DB 테이블 생성 (orders, order_items, payments)
2. 타입 정의
3. 주문/결제 서비스 레이어
4. 이용약관/개인정보처리방침 업데이트

#### Phase C: 결제 플로우 개발 (1~2주)

1. 주문서 작성 페이지 (배송지 입력, 주소 검색)
2. 토스페이먼츠 SDK 연동
3. 결제 승인 API Route (서버 검증)
4. 결제 성공/실패 페이지
5. 장바구니 → 주문 전환 로직

#### Phase D: 주문 관리 (1주)
