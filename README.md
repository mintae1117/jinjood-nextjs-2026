### 2. PG사(결제대행사) 선정 및 계약

#### 2-0. PG사란? (개념 정리)

- **PG(Payment Gateway, 결제대행사)**: 쇼핑몰과 카드사/은행 사이에서 결제를 중계하는 회사. 떡집이 카드사 수십 곳과 일일이 계약하지 않아도, PG사 한 곳과 계약하면 모든 카드/간편결제를 받을 수 있음.
- **결제의 돈 흐름**: 고객 카드 결제 → 카드사 승인 → **PG사가 대금 수령** → 수수료 차감 후 **정산일에 판매자(진주떡집) 계좌로 입금**.
- **두 가지 연동 방식**:
  - **단일 PG 직접 연동** (토스페이먼츠·나이스페이·KG이니시스 등): 한 PG사와 직접 계약·연동. 수수료가 명확하고 정산이 단순.
  - **통합 결제 솔루션** (포트원/구 아임포트): 여러 PG사를 하나의 SDK로 묶어주는 중계 레이어. PG사를 나중에 바꾸기 쉽지만, 별도 솔루션 이용료가 붙을 수 있음.
- 진주떡집처럼 **단일 매장·식품 판매** 규모에서는 **단일 PG 직접 연동(토스페이먼츠)** 이 가장 단순하고 비용 효율적.

#### 2-1. PG사 비교

| 항목 | 토스페이먼츠 ⭐추천 | KG이니시스 | 나이스페이먼츠 | 포트원(구 아임포트) |
|------|------------|---------|--------------|--------------|
| **유형** | 단일 PG | 단일 PG | 단일 PG | 통합 결제 솔루션 |
| **연동 난이도** | 쉬움 (한국어 SDK·문서 우수) | 보통 (전통적·문서 방대) | 보통 | 쉬움 (여러 PG 통합 SDK) |
| **Next.js 친화도** | 높음 (React SDK 공식 제공) | 보통 | 보통 | 높음 |
| **초기 비용** | 가입비·연회비 없음 | 가입비 협의(면제 多) | 가입비 협의 | 솔루션 이용료 별도 가능 |
| **카드 수수료(일반)** | 약 2.0~3.3% | 약 2.3~3.4% | 약 2.3~3.4% | 연동 PG 수수료 + α |
| **영세·중소 우대수수료** | 적용 가능 | 적용 가능 | 적용 가능 | 연동 PG에 따름 |
| **지원 결제수단** | 카드, 계좌이체, 가상계좌, 휴대폰, 카카오페이, 네이버페이, 토스페이 | 카드, 계좌이체, 가상계좌, 휴대폰, 간편결제 | 카드, 계좌이체, 가상계좌, 간편결제 | 연동 PG 범위 |

| **최소 요건** | 사업자등록 + 통신판매업 | 동일 | 동일 | 동일 |

> ※ 수수료율은 **카드사·결제수단·매출 규모·심사 결과에 따라 변동**되며, 위 수치는 계약 전 참고용 범위입니다. 정확한 요율은 계약 시 PG사가 제시하는 견적서를 기준으로 확정됩니다.

**추천: 토스페이먼츠** — Next.js용 React SDK(`@tosspayments/tosspayments-sdk`)를 공식 제공하고 한국어 문서/연동 예제가 가장 잘 정리되어 있어, 1인 개발·소규모 매장에 연동·유지보수 부담이 가장 적음. 가입비/연회비가 없어 초기 고정비도 0원.

#### 2-1-1. 결제수단별 수수료 (토스페이먼츠 기준 · 예시 범위)

PG 수수료는 "카드 한 종류"가 아니라 **결제수단마다 다릅니다.** 떡집 매출 구성에 따라 실효 수수료가 달라지므로 아래를 함께 검토하세요.

| 결제수단 | 예상 수수료 | 비고 |
|---------|-----------|------|
| **신용/체크카드** | 약 2.0~3.3% | 영세·중소 가맹점 우대수수료 적용 시 더 낮아질 수 있음 |
| **계좌이체** | 약 1.0~1.8% | 카드보다 저렴 |
| **가상계좌(무통장입금)** | 건당 약 300~500원(정액) | 비율이 아닌 **건당 고정 수수료** → 고액 주문(이바지·답례떡)에 유리 |
| **휴대폰 소액결제** | 약 3.5~5.5% | 수수료 높음 → 식품에는 비권장 |
| **카카오페이/네이버페이** | 약 2.0~3.5% | 간편결제사 수수료가 PG 수수료에 포함/별도일 수 있음 |

> **포인트**: 고객 단가가 높은 **이바지떡·답례떡·선물세트**는 가상계좌(정액 수수료)를, 단가 낮은 일반 떡은 카드/간편결제를 유도하면 전체 수수료를 줄일 수 있습니다.

#### 2-1-2. 영세·중소 가맹점 우대수수료

- 카드사는 연매출 구간에 따라 **우대수수료율**을 적용합니다(영세 가맹점일수록 낮음). PG 계약 시 사업자 매출 규모를 근거로 우대 적용을 요청할 수 있습니다.
- 신규 사업자는 매출 자료가 없어 초기에는 **일반 요율로 시작 → 매출 확정 후 우대 요율로 재산정**되는 경우가 일반적입니다.
- 정확한 적용 여부·요율은 카드사/PG사 심사 결과를 따릅니다. (계약 시 "우대수수료 적용 대상인지" 반드시 문의)

#### 2-1-3. 간편결제(카카오페이·네이버페이·토스페이) 도입

- 떡집 주 고객층(중·장년, 모바일 주문)을 고려하면 **간편결제 지원이 전환율에 큰 영향**을 줍니다.
- 토스페이먼츠 한 곳과 계약하면 카카오페이/네이버페이/토스페이를 **추가 개별 계약 없이** 결제창에서 함께 노출할 수 있어 편리합니다.
- 네이버페이 "주문형"은 별도 입점/심사가 필요할 수 있으니, 우선은 PG사 통합으로 제공되는 "결제형"부터 시작 권장.

#### 2-2. PG사 계약 시 준비 서류

- 정산 계좌 정보 (부산은행 112-2038-7604-08 / 정은아)

### 3. 법적 문서 업데이트

#### 3-1. 이용약관 업데이트 필요 사항

현재 `src/components/legal/TermsContent.tsx`에 결제 관련 조항(제8~11조)이 이미 존재하지만, 실제 결제 도입 시 구체화 필요:

- **제9조 (결제방법)**: 실제 사용할 PG사와 결제 수단 명시
- **제10조 (배송)**: 배송비 정책 구체화 (현재 3,000원 고정이면 명시)
- **제11조 (환불/교환)**: 환불 처리 절차와 소요 기간 구체화
- **추가 필요**: 주문 취소 가능 시점 (결제 후 ~ 배송 전), 부분 환불 정책

#### 3-2. 개인정보처리방침 업데이트 필요 사항

현재 `src/components/legal/PrivacyContent.tsx`에 기본 구조는 있으나:

- **수집 항목 추가**: 배송지 주소, 수령인 이름, 수령인 연락처
- **위탁업체 구체화**: 실제 계약한 PG사명, 배송업체명 기재 필수
  - 예: "토스페이먼츠 주식회사 - 결제 처리"
  - 예: "CJ대한통운 - 배송 업무"
- **보유기간**: 전자상거래법에 따라 결제 기록 5년, 배송 기록 5년 보관 의무 명시

#### 3-3. 결제 시 필수 고지 사항 (전자상거래법)

결제 페이지에 반드시 표시해야 할 내용:

- 총 결제 금액
- 결제 수단
- **"구매 조건 확인 및 결제에 동의합니다" 체크박스** (필수)
- 환불/교환 불가 사유 고지 (식품 특성)
- 주문자 정보와 수령인 정보

### 5. 결제 플로우 설계

```
[장바구니] → [주문서 작성] → [결제 요청] → [PG사 결제창] → [결제 완료] → [주문 확인]
                                    ↓                              ↓
                               [결제 실패]                    [결제 승인 API]
                                    ↓                         (서버 검증)
                               [에러 안내]                         ↓
                                                          [주문 상태 업데이트]
                                                                  ↓
                                                          [장바구니 비우기]
```

#### 5-1. 상세 단계

**Step 1: 주문서 작성 페이지** (`/checkout`)

- 장바구니에서 넘어온 상품 목록 표시
- 배송지 입력 (우편번호 검색 API - 다음 주소 검색)
- 수령인 정보 입력
- 배송 메모 선택/입력
- 결제 수단 선택
- 최종 금액 확인
- 약관 동의 체크박스

**Step 2: 결제 요청** (클라이언트)

- 주문 정보를 서버에 보내서 `orders` 레코드 생성 (status: pending)
- PG사 SDK로 결제창 호출

**Step 3: 결제 승인** (서버 - Next.js API Route 또는 Server Action)

- PG사로부터 받은 `paymentKey`, `orderId`, `amount`를 서버에서 검증
- 토스페이먼츠 승인 API 호출 (`POST /v1/payments/confirm`)
- **금액 위변조 검증 필수**: DB에 저장된 주문 금액과 PG사 응답 금액을 서버에서 비교
- 성공 시: `orders.status = 'paid'`, `payments` 레코드 생성
- 실패 시: `orders.status = 'failed'`, 에러 로깅

**Step 4: 결제 완료 페이지** (`/checkout/success`)

- 주문번호, 결제 금액, 배송 예정 안내
- 주문 내역 페이지 링크

**Step 5: 결제 실패 페이지** (`/checkout/fail`)

- 실패 사유 안내
- 재시도 또는 장바구니로 돌아가기

### 7. 보안 체크리스트

- [ ] **결제 금액 서버 검증**: 클라이언트가 보낸 금액을 절대 신뢰하지 않고, DB의 주문 금액과 PG사 응답 금액을 서버에서 비교
- [ ] **PG사 Secret Key**: 환경변수로만 관리, 클라이언트에 절대 노출 금지 (`TOSS_SECRET_KEY` - `NEXT_PUBLIC_` 접두사 없이)
- [ ] **웹훅 검증**: PG사 웹훅 호출 시 서명(signature) 검증
- [ ] **HTTPS 필수**: Vercel 배포 시 자동 적용
- [ ] **카드번호 비저장**: PG사 SDK가 처리하므로 직접 수집/저장 절대 금지 (PCI-DSS)
- [ ] **SQL Injection 방지**: Supabase 클라이언트 사용으로 기본 방어됨
- [ ] **주문 상태 변조 방지**: 클라이언트에서 직접 status 변경 불가하도록 RLS 설정

### 8. 세금 / 회계 관련

#### 8-1. 현금영수증

- 현금성 결제(계좌이체, 가상계좌) 시 **현금영수증 발행 의무** (소비자 요청 시)
- PG사에서 현금영수증 발행 기능을 제공하는 경우가 많음
- 미발행 시 가산세 부과 가능

#### 8-2. 세금계산서

- B2B 거래 시 세금계산서 발행 필요 (일반 소비자 대상이면 해당 없을 수 있음)
- 대량 주문(답례떡, 이바지떡) 시 사업자 고객이 요청할 수 있음

#### 8-3. 매출 신고

- PG사를 통한 결제는 자동으로 국세청에 보고됨
- 부가가치세 신고 시 PG사 매출 자료 활용

### 9. 배송 관련 고려사항

#### 9-1. 떡류 특수성

- **신선식품**이므로 당일/익일 배송 필수
- 배송 가능 지역 제한 필요 (전국 택배 가능한지, 부산 인근만 가능한지)
- 여름철 아이스팩/보냉 포장 고려
- **주문 마감 시간** 설정 (예: 당일 배송은 오전 10시까지 주문)

#### 9-2. 배송비 정책 확정 필요

- 현재 고정 3,000원으로 설정됨
- 지역별 차등 배송비 (제주/도서산간 추가)
- 퀵배송(부산 지역) vs 택배(타 지역) 구분 여부

#### 9-3. 택배사 연동 (선택)

- 초기에는 수동 운송장 입력으로 시작 가능
- 추후 CJ대한통운/롯데택배 API 연동 고려
